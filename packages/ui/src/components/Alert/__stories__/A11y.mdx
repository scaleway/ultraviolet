# Alert - Accessibility

## Description

A feedback component that displays short, important messages to attract user attention without interrupting tasks. Supports 5 sentiment types (danger, info, neutral, success, warning), an optional title, an action button, dismissible functionality, and two sizes.

## Summary

1. ✅ Perceivable: text contrast and presentation are correct, decorative icon is hidden from assistive technologies
2. ✅ Operable: controls are keyboard accessible via native `<Button>`, focus is restored to the previously focused element (or a consumer-provided `focusAfterClose` target) when the alert is dismissed
3. ✅ Understandable: content is properly readable, close button has a descriptive accessible name
4. ✅ Robust: `role="alert"`/`role="status"` is set on the container so alerts are announced by screen readers, title is programmatically associated via `aria-labelledby`

## ARIA Pattern

[ARIA Alert Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/alert/)

The pattern requires a `role="alert"` on the widget container so that dynamically rendered alerts are automatically announced by screen readers. The pattern also specifies that alerts must not affect keyboard focus (the [Alert Dialog Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/alertdialog/) is for situations where interrupting the workflow is necessary).

## Rules

Enforced by the component:

- ✅ [WCAG 1.3.1 - Info and Relationships (Level A)](https://www.w3.org/WAI/WCAG22/Understanding/info-and-relationships.html) - Title is rendered with a stronger text variant and content as a paragraph, preserving visual structure. The title is programmatically associated with the alert container via `aria-labelledby`.
- ✅ [WCAG 1.4.3 - Contrast (Minimum) (Level AA)](https://www.w3.org/WAI/WCAG22/Understanding/contrast-minimum.html) - Text uses sentiment colors from the theme that meet contrast requirements.
- ✅ [WCAG 1.4.11 - Non-text Contrast (Level AA)](https://www.w3.org/WAI/WCAG22/Understanding/non-text-contrast.html) - The sentiment icon and the 4px left border provide additional visual cues with sufficient contrast.
- ✅ [WCAG 2.1.1 - Keyboard (Level A)](https://www.w3.org/WAI/WCAG22/Understanding/keyboard.html) - The action button and close button use the native `<Button>` component, which is fully operable from the keyboard (Tab to move, Enter/Space to activate).
- ✅ [WCAG 2.1.2 - No Keyboard Trap (Level A)](https://www.w3.org/WAI/WCAG22/Understanding/no-keyboard-trap.html) - Focus can move in and out of the alert freely.
- ✅ [WCAG 2.4.3 - Focus Order (Level A)](https://www.w3.org/WAI/WCAG22/Understanding/focus-order.html) - Focus order follows the visual reading order: content, action button, close button.
- ✅ [WCAG 2.4.7 - Focus Visible (Level AA)](https://www.w3.org/WAI/WCAG22/Understanding/focus-visible.html) - The native `<Button>` component provides a visible focus indicator.
- ✅ [WCAG 2.5.3 - Label in Name (Level A)](https://www.w3.org/WAI/WCAG22/Understanding/label-in-name.html) - The close button has no visible text label (icon only), so this rule is not applicable; an `aria-label` is provided instead.
- ✅ [WCAG 3.2.1 - On Focus (Level A)](https://www.w3.org/WAI/WCAG22/Understanding/on-focus.html) - Receiving focus does not trigger any change of context.
- ✅ [WCAG 3.2.2 - On Input (Level A)](https://www.w3.org/WAI/WCAG22/Understanding/on-input.html) - No change of context is triggered by user input.
- ✅ [WCAG 4.1.2 - Name, Role, Value (Level A)](https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html) - The alert container exposes `role="alert"` (for `danger`/`warning` sentiments) or `role="status"` (for `info`/`neutral`/`success`), and the title is programmatically associated via `aria-labelledby`.
- ✅ [WCAG 4.1.3 - Status Messages (Level AA)](https://www.w3.org/WAI/WCAG22/Understanding/status-messages.html) - The alert exposes `role="alert"` (assertive) or `role="status"` (polite) based on sentiment, so status messages are programmatically determined and announced by assistive technologies when dynamically inserted.

To apply when using the component:

- [WCAG 1.4.1 - Use of Color (Level A)](https://www.w3.org/WAI/WCAG22/Understanding/use-of-color.html) - The sentiment is conveyed through color and an icon, but the icon is hidden from screen readers (`aria-hidden="true"`). Ensure the sentiment is also conveyed through text in the title or content (e.g., "Error: ...", "Success: ...") so that the meaning is not dependent on color alone.

## Accessibility issues

**Critical:**

- ✅ Resolved: Missing `role="alert"` on the alert container. The container now exposes `role="alert"` for `danger`/`warning` sentiments and `role="status"` for `info`/`neutral`/`success` sentiments, so screen readers announce the alert when it is dynamically inserted. This satisfies the [ARIA Alert Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/alert/) and WCAG 4.1.2 / 4.1.3.

**High:**

- ✅ Resolved: Focus management on dismiss. The close button now restores focus to the element that had focus before the close button received focus, or to a consumer-provided `focusAfterClose` ref when one is passed. Keyboard users no longer lose their place when dismissing the alert.

- ✅ Resolved: Title not programmatically associated. The title now has a stable `id` (via `useId`) and the alert container references it through `aria-labelledby`, so screen readers announce it as the alert's name.

**Medium:**

- ✅ Resolved: The close button `aria-label` is now descriptive. When a `title` is provided, the label becomes `Close {title} alert`; otherwise it stays `close`. Multiple alerts on the same page can now be distinguished by screen reader users.

- ✅ Resolved: The close button now uses `role="status"` (polite) for `info`/`neutral`/`success` sentiments and `role="alert"` (assertive) only for `danger`/`warning`, matching the urgency of each sentiment.

- ✅ Resolved: The `CloseIcon` inside the close button is now marked as `aria-hidden="true"`.

**Low:**

- ℹ️ Not implemented: An optional `role`/`politeness` override prop was not added since the sentiment-based defaults (`alert` for `danger`/`warning`, `status` otherwise) cover the common cases. Consumers can still control urgency by choosing the appropriate `sentiment`.

## Dependencies

- ✅ Resolved: Focus management on dismiss is handled internally (the previously focused element is tracked and refocused on close) and can also be driven by the consumer via the optional `focusAfterClose` ref prop.
